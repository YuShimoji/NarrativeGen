# Operator Workflow

人間オペレーターの実ワークフロー・痛点・品質目標を保持する正本。

## 全体フロー

5ステージパイプライン (SP-PIPE-001):

1. **Stage 1: 構想・設計** -- テキストエディタ/ホワイトボードでストーリー骨格を設計 (100% 手動)
2. **Stage 2: コンテンツ制作** -- Web Tester でノード・選択肢・条件・効果を制作 (80% 手動 / 20% ツール支援)
3. **Stage 3: 検証・調整** -- バリデーション + 推論エンジン + プレイテスト (30% 手動 / 70% 自動)
4. **Stage 4: エクスポート** -- JSON / CSV / Yarn / Ink / Twine に出力 (10% 手動)
5. **Stage 5: ランタイム統合** -- Unity Editor 等にモデルを配置・接続 (60% 手動)

1人で3つの帽子 (Writer / TechDesigner / Integrator) を切り替えて全ステージを担当する。

Stage 2-4 は Web Tester 内で完結する。Stage 1 はツール外、Stage 5 は Unity Editor 等。

## 工程ごとの痛点

- **Stage 1 → 2**: 設計 (ビート表等) から Web Tester への移行が手作業。ビート表がそのままノードにならない
- **Stage 2**: テキスト編集エリアが小さい。長文執筆には不向き (WritingPage 連携で解決予定だが延期中)
- **Stage 4**: Dynamic Text 構文が JSON 以外のフォーマットで失われる (JSON 主軸戦略で回避)
- **Stage 4 → 5**: エクスポートファイルの Unity プロジェクトへの配置が手作業

## 品質目標

- 全モデルが `npm run validate` でスキーマバリデーション通過
- 推論エンジンの到達可能性分析で孤立ノードがゼロ
- 主要ルートのプレイテストが完了済み
- E2E テスト: 0 failed を維持
- TODO / FIXME / HACK: 0 を維持

## Actor Boundaries

- **user**: Stage 1 全般 (構想・キャラクター設計・分岐ロジック設計)、Stage 2 テキスト執筆・分岐設計、Stage 3 プレイテスト目視確認・BGM/画像の感性確認、Stage 5 Unity UI 接続の設計判断、UI 方向性判断
- **assistant**: ドキュメント作成・同期、コード実装、バリデーション設計、エクスポーター実装、テスト修正
- **tool**: スキーマ検証 (自動)、構造検証 (自動)、到達可能性/パス/影響分析 (自動)、自動保存 (500ms debounce)
- **shared**: Stage 3 検証 (ツール自動 + 目視手動)、Stage 4 エクスポート (形式選択手動 + 変換自動)、次スライス選定、仕様策定

## 手動工程 / 自動化禁止工程

- ストーリー構想・テキスト執筆 (Stage 1-2 の創造的作業)
- プレイテスト目視確認 (Stage 3)
- BGM / 画像の感性確認 (Stage 3)
- Unity UI 接続の設計判断 (Stage 5)
- UI デザインの方向性判断

## 運用ルール

- 一度説明された workflow pain はここへ固定する
- 「本制作へ進む前に workflow proof が必要」な案件では、その proof 条件もここへ残す
