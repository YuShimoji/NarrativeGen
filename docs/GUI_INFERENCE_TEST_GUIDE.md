# 推論UI手動確認ガイド

## 前提条件

```bash
npm run dev
# → http://localhost:5173/
```

## 1. グラフハイライト確認

### 1.1 パスハイライト (ゴールド)

1. モデルを読み込む (quest.json が分岐が多く適している)
2. 「ノードグラフ」タブを開く
3. 任意のノードをクリックして選択する
4. 選択ノードへのスタートからの到達パス上にあるノード・エッジがゴールド (#d4a017) に変わることを確認

**確認済み**: 2026-03-17 動作確認OK

### 1.2 影響範囲ハイライト (コーラル)

**前提**: 選択ノードの選択肢に `effects` が設定されており、その effects が他ノードの `conditions` で参照されている必要がある。

**確認手順**:
1. GUI編集モードでノードを開く
2. 選択肢に効果を追加 (例: `setFlag` で `hasKey=true`)
3. 別ノードの選択肢に条件を追加 (例: `flag` で `hasKey=true`)
4. ノードグラフタブに戻り、効果を持つノードを選択
5. 条件参照先のノードがコーラル (#e07050) に変わることを確認

**注意**: サンプルモデル (quest.json 等) は effects/conditions の相互参照が少ないため、デフォルト状態ではコーラル表示が起きにくい。効果・条件を手動追加して確認する。

### 1.3 到達不能ノード半透明化

**前提**: セッション状態が設定されていること (プレイスルー中)。かつ、条件によりブロックされたノードが存在すること。

**確認手順**:
1. モデルを読み込む
2. 「プレイ」モードでストーリーを進める (選択肢を選んでセッション状態を作る)
3. ノードグラフタブに戻り、ノードを選択
4. セッション状態から到達不能なノードが opacity 0.4 で半透明になることを確認

**注意**: サンプルモデルは条件による分岐が少ないため、ほぼ全ノードが到達可能。到達不能ノードを確認するには、flag/resource 条件付き選択肢を含むモデルが必要。

## 2. デバッグクエリUI

### ノードID の見つけ方

ノードIDは以下で確認できる:
- **ノードグラフタブ**: 各ノードの矩形内に表示されているラベル (例: `start`, `quest_begin`, `shop`)
- **GUI編集モード**: ノードカードの上部に表示されるID
- **JSONエディタ**: モデルJSON内の `nodes` オブジェクトのキー
- **Live Preview**: 選択中ノードのID表示

### 確認手順

1. ノードグラフタブの推論パネル下部「クエリ」セクションを確認
2. テキストフィールドにノードIDを入力 (例: `quest_begin`)
3. 「分析」ボタンをクリック → 推論パネルの各セクションが更新される
4. グラフ上で該当ノードが選択状態になり、パスハイライトが適用される
5. 「クリア」ボタン → パネル内容がクリアされる (ノード選択状態は維持される — これは仕様)

## 3. 条件/効果エディタ

### 確認手順

1. GUI編集モードに切り替える (左上「GUI編集」ボタン)
2. 任意のノードカードを展開する
3. 選択肢を展開する (選択肢テキストの左の矢印をクリック)
4. 「条件」セクション内の「+追加」ボタンをクリック → 条件行が追加される
5. 条件タイプのドロップダウンで「アイテム所持」を選択
6. 名前フィールドにアイテムID (例: `sword`) を入力
7. 値フィールドに `true` (所持) または `false` (未所持) を入力

**効果エディタ**:
1. 「効果」セクション内の「+追加」ボタンをクリック → 効果行が追加される
2. 効果タイプのドロップダウンで「アイテム追加」または「アイテム削除」を選択
3. 名前フィールドにアイテムID (例: `sword`) を入力 (値フィールドは非表示)

### 既知の挙動

- **「+追加」でパネルが閉じる**: 条件/効果の「+追加」クリック時にノードカードの選択肢セクションが閉じてしまう場合がある。これは既存の GUI エディタの DOM 再レンダリングに起因する挙動であり、今回の Entity/Inventory 追加とは無関係。改善は別タスクとして扱う。

## 4. テスト用モデル推奨

推論ハイライトの全機能を確認するには、以下を含むモデルが望ましい:

- `setFlag` 効果 + `flag` 条件の相互参照 (コーラル確認用)
- 複数パスの分岐 (パスハイライト確認用)
- flag/resource 条件でブロックされるノード (到達不能確認用)

サンプルモデルでは `quest.json` が最も分岐が多いが、effects/conditions の相互参照は限定的。
専用の推論テストモデルの作成を検討する。
